前端安全学习记录

前端安全的背景和重要性
随着互联网技术的飞速发展，我们逐渐迈入了信息时代。然而，随之而来的是用户数据泄露和隐私泄露的风险。前端安全成为我们的首要任务。为了保护用户数据，确保用户的隐私不受侵犯，我们必须加强前端安全。
前端安全常见的问题

  • XSS（跨站脚本攻击 英语：Cross-Site Scripting）
  • CSRF（跨站请求伪造 英语：Cross-site request forgery）
  • 点击劫持
  • 数据传输安全
  • 登录暴力破解
  • 三方开源组件
  • https 的问题
  • Iframe
  • CDN 劫持

XSS（跨站脚本攻击）
○ 概念：攻击者往目标网页中注入恶意脚本，通过用户浏览器执行，从而攻击网站，或者获取用户的敏感信息。demo1
○ 常见的实现方式：
■ eval: 执行恶意脚本
■ document.write：使用 document.write 方法将恶意脚本代码直接输出到页面中。（document.write 和innerHTML 的区别）
■ appendChild：使用 DOM 的 appendChild 方法向页面中插入新的元素节点，其中包含恶意脚本代码。
■ location.href：通过修改网址的参数字符串，将恶意脚本代码作为参数传递给另一个页面，该页面将脚本插入页面并运行。
■ XMLHttpRequest：通过 AJAX 请求将恶意脚本代码发送到服务器，再从服务器获取响应后执行。
■ Cookie：通过设置恶意脚本代码作为 Cookie 的值，在下一次请求时将其发送到服务器并执行。
■ 页面中随意引入三方脚本（三方脚本中可以偷偷获取用户的 cookie，获取用户信息）
○ 场景思考：
■ URL 中放入接口地址，前端在代码中执行该接口，是否存在安全风险？
○ 如何避免：
■ 使用 HTML 转义。对外部插入的内容要永远保持警惕。 对所有外部插入的代码都应该做一次转义，将script,& < > “ ‘ /等危险字符做过滤和转义替换，同时尽量避免使用 innerHTML,document.write,outerHTML,eval等方法，用安全性更高的 textContent,setAttribute 等方法做替代；
■ 开启 CSP 防护。内容安全策略（CSP）的设计就是为了防御 XSS 攻击的，通过在 HTTP 头部中设置 Content-Security-Policy,就可以配置该策略，如果将 CSP 设置成一下模式：Content-Security-Policy: script-src ‘self’
● 影响：（不推荐）
○ 不允许内联脚本执行;
○ 禁止加载外域代码;
○ 禁止外域提交;
■ 设置 HttpOnly。当然这已经是属于降低 XSS 危害的方法，对于所有包含敏感信息的 cookie，都应该在服务端对其设置 httpOnly，被设置了 httpOnly 的 cookie 字段无法通过 JS 获取，也就降低了 XSS 攻击时用户凭据隐私泄漏的风险。

CSRF（跨站请求伪造）
○概念：攻击者利用用户当前的登录状态，在用户不知情的情况下，以用户的名义发送恶意请求。
○ 典型案例

○ 场景思考：恶意网站向目标网站以受害者身份执行了一些带权限的操作，给受害者带来一系列危害和损失。
○ 如何避免：
■ SameSite Cookie 如果能让第三方发起的请求无法携带相关的 cookie，那一切问题好像就解决了啊。值得庆幸的是，浏览器针对 cookie 提供了的属性，该属性表示 Cookie 不随着跨域请求发送。
■ CSRF Token：在用户访问网站时，后台服务器根据算法再生成一个 Token，然后把这个 Token 放在 seesion 中，当网站发起请求时，不仅要携带 cookie 凭证，还要把这个 Token 也带上，后台一起校验之后确认身份再执行操作。由于 SCRF Token 是放在 session 中，因此当第三方网站发起请求时，无法拿到这个 SCRF Token，故身份校验不再通过，就达到了防御攻击的效果。
■ 同源检测：和是 http 请求的头部字段之一，用来标志该请求是从哪个页面链接过来的。因此后台服务器可以通过检查该字段是否是来自自己的网站链接，来避免第三方网站发起 CSRF 攻击。但是同源检测的可靠性并不高，比如在 302 重定向的时候，为了保护来源，http 请求不会携带字段，而字段会受到 Referer Policy 规则的限制而不发送。
■ 增加二次验证：针对一些有危险性的请求操作（比如删除账号，提现转账）我们可以增加用户的二次，比如发起手机或者邮箱验证码检验，进而降低 CSRF 打来的危害。

点击劫持
○ 概念：攻击者伪造合法网页将恶意网页覆盖在诱导用户点击的合法网页上，实现恶意操作。
○ 场景思考：
■ 钓鱼攻击：攻击者通过伪装成合法网站或服务，将恶意内容隐藏在看似无害的链接或按钮上，引诱用户点击，以获取用户的敏感信息。
■ 窃取用户登录信息：攻击者使用透明的框架覆盖在正常的登录页面上，用户误以为是正常的登录界面，输入账号密码后，实际上是被攻击者所接收。
■ 窃取用户隐私信息：攻击者将恶意代码嵌入在网页的某个元素上，这个元素的位置与一个常用的功能按钮或表单重叠，当用户点击该按钮或表单时，实际上点击了恶意代码，从而窃取用户的隐私信息。
○ 如何避免：
■ 使用脚本来防御：可以使用 JavaScript 来防范点击劫持，比如通过检查是否在顶层窗口打开页面、使用脚本检查页面是否被嵌入到 iframe 中等。
■ 提高用户安全意识：教育用户有关点击劫持的风险，以便他们能够识别和避免点击劫持的攻击。
数据传输安全
○ 概念：使用 HTTP 传输的数据容易被窃取和篡改。
○ 典型案例：
■ 未加密敏感信息传输：使用 HTTP 传输的数据，如用户名、密码、信用卡号等敏感信息，如果未经过适当的加密保护，被窃取和破解。
○ 如何避免：
■ 使用 HTTPS 协议：HTTPS 通过使用 SSL/TLS 协议对传输的数据进行加密，可以防止数据在传输过程中被窃取和篡改。
■ 避免使用明文密码和敏感信息。
■ 使用 jsencrypt 对密码 手机号码登信息进行加密
登录暴力破解
○ 概念：通过多次尝试不同的密码来强行进入受保护的账户或系统
○ 常见实现方式
■ Brute Force Attack（暴力破解攻击）：攻击者通过尝试所有可能的用户名和密码组合来登录目标账户。这通常会使用自动化程序来加速尝试过程。
■ Dictionary Attack（字典攻击）：攻击者使用预先准备好的密码字典来尝试登录目标账户。密码字典包含常见的用户名和密码组合，以及其他可能的组合。
■ Credential Stuffing（凭证填充）：攻击者使用从一个被泄露的网站获得的用户名和密码组合来尝试登录其他网站账户。许多人在不同的网站上使用相同的用户名和密码，使得凭证填充攻击成为可能。
■ Password Spraying（密码喷洒）：攻击者针对目标系统使用少量的常见用户名，然后对每个用户名尝试一组预先选择好的常见密码。这种攻击方式减少了暴力破解的尝试次数，从而更不容易被检测出来。
○ 如何避免：
■ 使用强密码：选择一个复杂、不易被猜测的密码。使用组合字符、数字和特殊符号，并确保密码长度较长。避免使用常见的短密码，比如”123456”或”password”。
■ 启用两步验证（2FA）：启用两步验证可以为登录提供额外的安全层。这需要用户在输入用户名和密码之后，再通过其他方式（如手机验证码、指纹识别等）进行身份验证。
■ 限制登录尝试次数：通过设置登录尝试次数上限和登录锁定时间，在黑客进行多次尝试之后暂时锁定账户。这样可以防止暴力破解攻击。
三方开源组件
○ 概念：是指在软件开发过程中，使用的第三方开源组件可能存在安全漏洞或风险，可能会导致整个系统的安全受到威胁。
○ 典型案例：Antd 圣诞失控事件也正是源自于此，开发者别出心裁地在框架代码中埋下了一个会在圣诞节当天触发的彩蛋：一个不曾提前告知、无法手动禁止、并且“潜伏”了 106 天之久的按钮“惊喜”。
○ 场景思考：使用三方组件的时候要尽量谨慎。
○ 如何防御：手动检查这些第三方代码有没有安全问题是个苦差事，主要是因为应用依赖的这些组件数量众多，手工检查太耗时，好在有自动化的工具可以使用，比如：

Https 的问题
○ 为了保护信息在传输过程中不被泄露，保证传输安全，使用 TLS 或者通俗的讲使用 HTTPS 已经是当今的标准配置了。然而事情并没有这么简单，即使是服务器端开启了 HTTPS，也还是存在安全隐患，黑客可以利用 SSL Stripping这种攻击手段，强制让 HTTPS 降级回 HTTP，从而继续进行中间人攻击。问题的本质在于浏览器发出去的第一次请求就被攻击者拦截了下来并做了修改，根本不给浏览器和服务器进行 HTTPS 通信的机会。大致过程如下，用户在浏览器里输入 URL 的时候往往不是从 https://开始的，而是直接从域名开始输入，随后浏览器向服务器发起HTTP 通信，然而由于攻击者的存在，它把服务器端返回的跳转到 HTTPS 页面的响应拦截了，并且代替客户端和服务器端进行后续的通信。由于这一切都是暗中进行的，所以使用前端应用的用户对此毫无察觉。
○ 如何防御：办法是使用 HSTS（HTTP Strict Transport Security）nginx 配置 add_header Strict-Transport-Security max-age=31536000

CDN 劫持
○ CDN 劫持是指攻击者利用某种手段窃取或篡改内容分发网络（CDN）的域名解析，将用户的请求重定向到恶意的服务器上。这种攻击可以导致用户访问到虚假的网站、下载到恶意软件或者泄露敏感信息。
○ 前端可以做的：
■ SRI 全称 Subresource Integrity - 子资源完整性，是指浏览器通过验证资源的完整性（通常从CDN 获取）来判断其是否被篡改的安全特性。通过给 link 标签或者 script 标签增加 integrity属性即可开启 SRI 功能，比如 <script type=”text/javascript” src=”//s.url.cn/xxxx/aaa.js”integrity=”sha256-xxx sha384-yyy” crossorigin=”anonymous”>
■ 开启 SRI 能有效保证页面引用资源的完整性，避免恶意代码执行
■ 浏览器如何处理 SRI
● 当浏览器在 script 或者 link 标签中遇到 integrity 属性之后，会在执行脚本或者应用样式表之前对比所加载文件的哈希值和期望的哈希值。
● 当脚本或者样式表的哈希值和期望的不一致时，浏览器必须拒绝执行脚本或者应用样式表，并且必须返回一个网络错误说明获得脚本或样式表失败。
○ CDN 劫持场景比较复杂，前端能做的比较有限，最终要的还是从服务器端根本解决劫持的问题。
○ 网站管理员可以采取以下措施：
■ 使用 HTTPS 协议：通过使用 HTTPS 协议，可以加密通信，防止被中间人劫持或篡改数据。
■ 增强域名解析的安全性：采用安全的域名解析服务，如 DNSSEC（Domain Name System SecurityExtensions），可以提供域名解析的完整性和身份验证。
■ 监控域名解析配置：定期审查域名解析的配置，发现异常情况及时处理。特别关注域名解析的TTL（Time to Live）值，以防止其被非法修改。
■ 限制 CDN 节点访问权限：只允许特定的 IP 地址或特定的网络访问 CDN 节点，以减少攻击面。
■ 进行安全测试：定期进行安全测试，检测 CDN 的安全漏洞和配置问题，及时修复。总结：前端安全是保护前端代码和用户数据安全的重要措施，需要在开发中加入相应的防御措施并不断更新和完善，以保障系统的安全性。
